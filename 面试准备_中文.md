# 午餐 Agent · 面试准备（中文，给我自己看）

> 这份文档是我自己面试前看的，用大白话写。代码细节看 `README.md` 和 `SKILL.md`，这里只讲：**这个东西是什么、为什么这么设计、demo 怎么跑、可能被问到什么。**

---

## 一句话讲清楚这个产品

一个帮你「自动点午餐」的 agent。你只配置一次偏好，之后每个工作日中午前，它自己去找附近餐厅、筛掉不安全/超预算的、排个序、挑一个，然后在 Discord 上发一条消息让你确认。你回 `1` 就定，回 `2` 换一个，回 `3` 今天不吃。

**北极星：你永远不会因为「忘了点」或「懒得选」而饿肚子。** 这个 agent 本身绝对不能成为你错过午餐的新原因。

---

## 它解决的真实问题

目标用户是忙的上班族，午餐经常吃不上，原因就两个：

1. **忘了点** —— 一忙就过了饭点
2. **懒得选** —— 打开外卖 app，几十个选择，决策疲劳，干脆不点了

这两个的结局都一样：没饭吃。我的 agent 把这两个原因都干掉了：到点自动提醒 + 它帮你选好一个，你只要说「行 / 换 / 不」。

## 核心设计理念（面试重点，能讲出「为什么」很加分）

### 1. Agent 是「替你做」，不是「提醒你做」
只会提醒的不叫 agent，叫闹钟。真正的价值是它直接行动。为了安全，每个订单都卡在硬性限制内，所以最坏情况只是「点了个你不是特别想吃的」，而不是「点了会过敏的 / 超预算的」。

### 2. 用「排除法」帮你，而不是「让你选」
决策疲劳的核心是「选项太多」。所以我不把菜单丢给你，而是 **帮你排除掉大部分，最后只给你一个**。你只需要「反应」，不需要「挑选」。

### 3. 硬限制 vs 软偏好（这是整个系统的骨架）
- **硬限制（代码强制，绝不破例）**：过敏原、预算上限、绝对不吃的（no-go）。过敏信息确认不了，这个菜直接跳过。
- **软偏好（只影响排序，不会排除）**：菜系喜好、不太爱吃的、最爱的、距离、历史评分、最近吃过没。

这样设计的好处：**安全的事交给规则，口味的事交给排序。** 出了岔子也不会越界。

### 4. 排序看多个信号（不是只看一个）
筛完之后，剩下的候选按这些打分排序，权重从大到小大概是：

- **距离最重要**：200 米内 +20，400 米内 +12，600 米内 +6
- **你的最爱 / 历史点赞**：各 +25
- **Google 评分**：≥4.5 加 10，≥4.0 加 5
- **菜系喜好**：最多只加 12 分（**故意压低**，下面解释）
- **预算内**：+3
- **扣分项**：不爱吃的 -8、最近吃过同一家/同菜系会扣分（避免天天重复）

> ⚠️ **菜系为什么只算「轻微加分」？** 这是我后来专门改的。一开始菜系权重很高（+50），结果它老推日料，因为我把日料排第一。但实际上 **附近不一定有那么多某类餐厅**。所以我把菜系降成「轻微的打破平局项」，让 **距离、评分、预算** 来主导。这样换个地点、换个人用，推荐都合理。这点如果被问到「你怎么权衡用户偏好和现实约束」，是个很好的回答。

### 5. 信任是「越来越准、越问越少」，不是「越来越能乱花钱」
我没有做那种「用久了就给你更高额度」的权限爬坡。信任的意思是：它推荐得更准、打扰你更少。通过简单的反馈闭环（点赞/拉黑/每周小结）实现，不需要复杂的学习模型。

### 6. 不依赖大模型也能跑
筛选、排序、确认、下单、失败处理 **全是确定性逻辑**。大模型只在「读乱七八糟的菜单文字、判断过敏原」时可选地帮忙，而且它只能 **让检查更严**，绝不能把一个关键词判定为「不安全」的结果翻回「安全」。这点对安全很关键。

## 真实数据从哪来（这次 demo 的亮点）

CEO 之前觉得太简单，所以我接了 **真实的数据源**，让 demo 是 live 的：

- **OpenStreetMap / Overpass API**：免费、不要 key，用来发现附近餐厅、查营业时间。
- **Google Places API**：补充 OSM 没有的东西 —— 人均价格档位（$~$$$$）、星级评分、评价数、地图链接。
- **DoorDash 深链**：我们不碰支付。确认后给一个 DoorDash 搜索链接，你自己点进去手动下单。**这是诚实的设计**：没有公开的下单 API，硬做浏览器自动化既脆弱又不安全，所以做成「人工交接」。

> 三步发现流程：① 找附近餐厅 → ② 按营业时间过滤掉关门的 → ③ 用 Google 补充评分/价格，再交给排序。

## Demo 怎么跑（明天现场）

**位置**：Bellevue Key Center（已配在 `.env` 里），时区是太平洋时间 PT。

一条命令搞定：

```bash
pkill -f "agent.py --demo"; python src/agent.py --demo
```

- `pkill` 那半句是先清掉可能残留的进程（**重要**，下面踩过的坑会讲）。
- `--demo` 表示「假装到点了，立刻跑一轮 live」，不用真等到中午 12 点。

跑起来后，Discord 会收到 **一条** 卡片，长这样：

```
Hey! 👋 Lunch pick for you

🔎 Discovery complete
Checked 144 nearby places · 126 viable now · filtered 18 closed
Next: ranking by distance, price, rating, and your preferences.

🍽️ Earls · lunch in ~5 min
📍 349 m   ·   ⭐ 4.7 (7490)
💵 ~$20/person
✨ cuisine you like · close by · highly rated

🛒 Order on DoorDash: https://...
🗺️ Map: https://...

1️⃣ order   ·   2️⃣ another   ·   3️⃣ skip
```

在 Discord 回：
- `1` = 就它了（回完就结束，不会再多发消息）
- `2` = 换一个（立刻换，不用等）
- `3` = 今天不吃

## 这次我打磨的几个 UX 细节（被问「你怎么做产品」可以讲）

1. **从「丑」到「清爽」**：原来卡片把内部打分（`cuisine rank +50` 这种 debug 信息）直接给用户看，还有一堆「unknown / 营业时间 / 100 多家候选」。我把这些砍了，只留用户真正要看的：**距离、评分、价格、为什么推荐**，用 emoji 分层，一眼能看懂。

2. **「为什么推荐」说人话**：内部是 `distance <=400m +12`，给用户看的是 `close by`、`well rated`、`cuisine you like`。

3. **价格不冗余**：原来同时显示 `$$` 和 `~$20/person`，重复。现在只留 `~$20/person`（人均金额对做预算更有用）。

4. **一条消息，不刷屏**：原来发现摘要和推荐卡片是两条，确认后又多一条「✅ Selected」。我合并成一条卡片，确认后不再多发 —— 因为「你回 `1`」本身就是确认，没必要再 ping 你。

5. **DoorDash 链接修对了**：之前链接点开跳不到搜索结果，原因是少了 `?event_type=search` 参数，补上就好了。

## 踩过的坑（如果被问「遇到什么问题」）

- **Discord 发两条**：不是 bug，是 **同时跑了两个 demo 进程**，都连同一个频道，各发各的。教训：现场只跑一个，跑前先 `pkill`。
- **测试不稳定**：有几个测试依赖「当前时间」和 `.env` 里的真实 key，晚上跑会把样例店判成关门、或误触发真实网络请求。我加了固定时间 + 隔离 fixture，让测试 **确定性、离线、可重复**。现在 **75 个测试全过，0.07 秒**。

## 故意没做的（边界要清晰，面试官喜欢「知道自己不做什么」）

| 没做 | 为什么 |
|------|--------|
| 真实支付 | 假设平台已配好，是另一个独立问题 |
| 真·下单 API | 没有靠谱的公开 API；做成 DoorDash 深链人工交接，诚实且不脆弱 |
| 偏好学习模型 | 排序 + 点赞/拉黑 + 每周小结已经体现信任闭环 |
| 完整 dashboard | CLI + JSON 状态够用 |
| 日历/天气/定位信号 | 集成成本和隐私成本高；暂停/跳过已经覆盖「今天不点」 |
| 推荐没吃过的新东西 | 先建立信任，先待在用户说过的偏好里 |

## 一句话收尾（面试可以这么总结）

> 「这个项目我想证明的不是『我能接多少 API』，而是 **agent 的判断力**：在硬性安全约束下，用多信号排序帮用户做决定，用最少的打扰拿到一个安全、合预算、靠谱的午餐。真实数据（OSM + Google）让它不是玩具，DoorDash 深链让它诚实，UX 上我反复打磨到『一眼看懂、一键确认』。」
