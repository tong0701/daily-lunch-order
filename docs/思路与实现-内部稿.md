# 每日自动点餐 Agent — 思路与实现

> 内部稿。记录两层东西：每个设计判断当时是怎么想的，以及它在代码里最终长成了什么样。
> 用来：(1) 我们自己把项目想清楚；(2) 作为 live presentation 的底稿。

---

## 0. 先搞清楚这道题在考什么

没有上来就写代码，先判断题目核心。

表面是"做一个自动点外卖的机器人"，但现实里没有能直接用的外卖开放接口，硬对接真实平台没有意义。这道题真正想看的是 **agent 的判断逻辑、出错怎么处理、怎么慢慢让用户信任它**——不是考第三方集成能力。

**这个判断直接决定了架构：**
- 下单这一层用一个 `MockProvider` 顶着，藏在 `Provider` 抽象接口（`search_menu / check_availability / place_order / get_order_status`）后面，以后换真实平台不动主流程；
- 绝大部分精力放在决策和异常逻辑上；
- 全程离线可跑，支持"用大模型"和"不用大模型"两种模式，保证 demo 一定能跑通。

> 落地：`src/provider.py` 定义接口 + `MockProvider`（17 道菜，故意埋了各种边界数据，见 §3）。

---

## 1. 服务对象与核心目标

不做"谁都能用"的通用工具，专门针对一个具体的人：工作日很忙、经常忘了点午饭，或者纠结太久到最后没吃上的上班族。

痛点只有两个，且指向同一个结果——**没饭吃**：
1. 太忙，忘了点；
2. 纠结太久，最后没吃上。

由此提炼出核心：

> **中午绝不饿着，而且不用为这事动脑子。**

后面每个决策都拿这句话来量。它还隐含一条硬要求：**agent 自己绝不能变成新的"让用户没饭吃"的原因**——这给出错处理定了很高的标准。

---

## 2. 几个关键取舍

### 2.1 要不要自动下单（最难的一个）

- 支持：只有真替你下单，才解决得了"忘了点"。只发提醒 = 闹钟，谈不上自动化。
- 反对：没经过明确同意就花了钱。

**最后决定自动下单成立**，理由是：每一单都卡在两道硬限制里（安全、预算），最坏也只是一份不合口味但安全、不超预算的饭。**"安全"是靠硬限制卡死的，不是靠"干脆不下单"保证的。**

> ⚠️ 已知薄弱点（要在现场答好）：这条辩护默认"不想要还能手动取消"。但外卖一旦餐厅接单，很多平台取消不了。所以更稳的版本是：auto-order 是 onboarding 里显式开的 opt-in，且只在"沉默"时触发；如果重做，会考虑把 auto-order 设成"N 次成功确认后才解锁"——也就是 §2.3 里我砍掉的信任爬坡，只用在这一个权限上。

### 2.2 硬限制 vs 软偏好（整个设计的地基）

把用户输入分成两类：

- **硬限制**（过敏、预算）：在代码里强制执行，模型说了不算。不能确认是否含过敏原 → 直接当不安全跳过，不赌。
- **软偏好**（菜系排序、不喜欢、收藏）：只用来调推荐顺序，不直接排除。

分清这两类，才能同时避免两个失败：出不安全的单，或约束太死点不出任何东西。

> 落地（`src/agent.py`）：
> - **硬过滤** `hard_filter_item()`：超 `ceiling = max_usd + approved_overage_usd` 的删；命中硬过敏的删；命中 `no_go`（含用户拉黑的 `do_not_show_again`）的删；`allergen_confirmed=False` 的删；非 `main`（配菜/饮料）的删；命中 `fallback_foods` 的删（保留给失败阶梯，不进日常池）。
> - **软排序** `score_item()`：菜系排名 `(菜系总数 − 名次) × 10`；命中 favorite `+25`；命中 dislike 每个 `−8`；近 5 天点过 `−20`（保证不天天一样）；价格在基础预算 `max_usd` 内 `+3`。
> - **选择** `pick_option()`：从分数前三名里，按分数**加权随机**挑一个（不永远选第一）。

### 2.3 "信任爬坡"最后放弃了

一开始想：刚用时多让你确认，用久了再放开，权限慢慢挣。听起来很贴"随时间建立信任"。

最后砍掉，理由：就一顿十几块、有硬限制兜底、不满意能取消的午饭，专门搞一套分级权限是过度设计。

我把"长期信任"**重新定义**成：agent 越来越懂你、推得越来越准、问得越来越少——而不是"越来越敢替你做主"。

> 注意：这条和 §2.1 的薄弱点相关。为什么这里砍爬坡是对的，但放在 auto-order 的 consent 风险上又值得保留，把这个张力讲清楚。

### 2.4 确认流程

午饭前约 `lead_time_min`（默认 5 分钟）推**一道**菜，四条路径：

| 用户操作 | 系统反应 |
|---|---|
| order_it | 下单 |
| show_another | **立刻**重挑（人在线，不让你等） |
| not_today | 记 skip，今天结束 |
| 无响应 | 等 `reminder_interval_min`（10 分钟）再提醒一次；仍无响应且 `auto_order_on_no_response=true` → 自动下单 |

**关键判断：主动说"不"和"沉默"区别对待。** 主动点 show_another/not_today 说明人在线，立刻响应；10 分钟的等待只用于沉默。保底餐不单独搞流程，作为标注过的普通候选出现在同一次确认里。

## 3. 出错处理（这部分是重点）

总原则：**出错时，agent 的任务不是不惜代价把单下成，而是要么还给你弄到一口安全的饭，要么早点告诉你，让你来得及自己解决。**

**失败阶梯：**
1. 找不到合适的 → 上 `fallback_foods`（仅 main）→ 连保底都不行 → **提前通知**你自己点，记 `failed`；
2. 下单提交失败 → 换下一个候选 → 保底 → 通知；
3. 下了但状态未知 → **绝不重下**（防双重扣费）→ 记 `needs-verification`，让你去 App 核对。

从用户角度，"下单失败"和"结果不确定"是一类事：今天没替你搞定，你要额外处理一下。

**三条铁律（写死在代码里）：**
1. 绝不为下单成功破硬限制；
2. 绝不闷声失败或拖太晚；
3. 绝不重复扣费。

> 落地：
> - `place_with_failure_ladder()` 处理路径 2/3；`handle_no_valid_option()` 处理路径 1。
> - 防重复扣费靠**每日幂等键** `lunch-order-YYYY-MM-DD`：同一天同一个 key 不会二次下单；且 guard 在 `state.json` 记录今天为 `ordered/skipped/failed/needs-verification` 任一时直接退出。
> - `MockProvider` 支持 `fail_item_ids` 和 `unknown_status_item_ids`，专门用来在测试里制造"下单失败""状态未知"。

## 4. 哪些功能主动没做，为什么

48 小时，深度优先于广度，原则：**用最便宜的办法解决真实需求，不把产品做成另一个东西。**

| 没做的 | 理由 |
|---|---|
| 真实支付 | 默认已绑定；开通支付是另一个问题，展示不了更多 agent |
| 对接真实外卖平台 | 没有开源接口，用 mock，精力留给逻辑 |
| 偏好学习模型、完整 dashboard | 加权选择 + 每周总结已够展示闭环 |
| 读日历 / 定位 / 天气 | 隐私 + 集成成本高，"什么时候别点"用 skip/暂停覆盖 |
| 推新尝鲜 | 产品立在信任上，一次踩雷就掉信任，只在偏好范围内选 |
| 配送 / 退款 / 售后 | agent 职责到"一个好单、下成并确认"为止，后面归平台 |
| 健康 / 饮食目标 | 那是另一款产品，加进来把定位做糊 |
| 真实确认通道 | 状态机已设计、用 `--response` 模拟；生产入口目前直接 auto-order，缺真实 push/SMS/chat + inbound 回调 + 超时任务 |
| 子串匹配 | demo 够用；泛词会误伤，下一步换结构化 tag |

## 5. 信任闭环

- 吃完一键 👍/👎（`--rate`）；
- 每周一条总结（`--summary`：点了啥、最高分、"别再出现"列表）；
- "别再出现"（`--block`）直接进 `no_go` 黑名单。

用得越久越贴口味，用户**看得见**它在变好，信任就这么攒起来。更深的学习留作后续。

> 落地：评分、do-not-show-again、每日历史都存在 `data/state.json`；`--block` 同时写入 state 和 `user_preferences.yaml` 的 `no_go`。

## 6. 开发顺序

全程一个原则：**逻辑在前，体验在后。** 

1. 先读懂题、找准考点（§0）；
2. 把选菜、报错、信任的逻辑想透、能讲清（§1–§5）；
3. 再做体验：友好的 onboarding、专业克制的口气、把引擎 trace 藏到 `--verbose` 后面；
4. 最后标准化：写测试、完善文档、整理 repo，对齐作业的交付要求。

一路在压"过度设计"：一个方案听起来聪明但只加复杂度、不多展示判断，就砍掉或留作以后。简单、稳定、可复现，胜过功能堆满。

## 7. 跑起来才发现的问题（迭代）

光想会有漏洞，实际运行暴露了两处：

1. **味噌汤 bug**：第一次 demo，agent 推了一碗 $4.5 的味噌汤当午饭——违背北极星（你会饿着）。根因是排序没分正餐和配菜。修法：给每道菜加 `item_type`（main/side/drink），**只从 main 里选**。
2. **fallback 测试 bug**：写"保底被启用"的测试时报错，查下来是测试自己把保底也屏蔽了，agent 的行为其实是对的（连保底都没了就通知你自己点）。这个用例顺带逼我把"**保底算不算日常候选**"定清楚：不算——`build_candidates()` 里把命中 `fallback_foods` 的菜标成 "reserved for fallback ladder" 排除掉，让日常池可以为空、而保底仍然可用。

## 8. 现有方案的不足与下一步（自己先点破）

**1. 过敏原校验改成 fail-closed（最重要，已修复）**
之前开 LLM 后，模型说 safe 就放行，等于把安全判断权交给模型，和"安全必须靠代码"自相矛盾。

> 现在的实现（`llm_match_allergens`）：先跑关键词检查，判 unsafe 立刻返回、**不调模型**（也保住 `allergen_confirmed` 兜底）；只有关键词判 safe 才调模型复核；模型 UNSAFE → unsafe，SAFE → safe；没有 API key / 没装 openai / 调用报错 → 回退到关键词的 safe 结果。**净规则：只有 keyword-safe 且 (llm-safe 或 llm 不可用) 才放行。模型只能收紧、永远推不翻代码底线。**
> 两个测试钉死：① 强制模型返回 SAFE，含 shellfish 的菜仍被 drop；② 模型返回 UNSAFE，本来 keyword-safe 的菜被 drop。

**2. 用户确认目前是模拟的**
推送 → 等待 → 自动下单的时序在设计上完整，代码里用的是立即返回的模拟值，没有真实异步等待和回复处理。合理的范围取舍，已在 README 标注。下一步：真实 push/SMS/chat 出站 + inbound 回调 + 状态持久化 + 超时任务。

**3. 菜品匹配较粗糙**
偏好/黑名单现在靠子串匹配，泛词易误伤（比如 "wrap" 会命中所有卷类）。下一步换结构化 tag 匹配。

## 附. 代码结构与测试

**模块职责：**

| 文件 | 职责 |
|---|---|
| `src/agent.py` | 确定性主流程：guard → 硬过滤 → 软排序 → 确认 → 下单 → 失败阶梯 → 记录 → 信任闭环 CLI |
| `src/config.py` | 加载时校验 yaml；缺字段或预算过低（<$5）给清晰报错，不崩 |
| `src/provider.py` | `Provider` 接口 + `MockProvider`（17 道菜，含边界数据） |
| `src/menu_matcher.py` | 关键词过敏原匹配；可选 LLM（fail-closed） |
| `src/onboard.py` | 交互式 onboarding，写 yaml，过敏原复述确认 |
| `src/scheduler.py` | 轮询时钟，在 `lead_time` 前触发 agent |
| `data/state.json` | 每日状态（幂等）、历史、评分、do-not-show-again |

**MockProvider 里故意埋的边界（demo 现场最有杀伤力的一段）：**
- Shrimp Tempura（shellfish）、Pad Thai with Peanuts（peanuts）→ 硬过敏，drop
- Mystery Chef Special（`allergen_confirmed=False`）→ 信息不全，drop
- Premium Wagyu $24 > 上限 $21 → 超预算，drop
- Liver and Onions → 命中 no_go，drop
- Miso Soup / Fruit Cup / Iced Green Tea → 非正餐，drop
- 跑 `python src/agent.py --once --verbose` 可看到 17 道菜被逐条 KEEP/DROP + 打分排序。

**14 个离线测试（钉住安全与失败声明）：**
- 配置校验 ×3、硬过滤 ×4（硬过敏 / 未确认 / 配菜饮料 / 不超 ceiling）、每日幂等 ×1、失败阶梯 ×3（用保底 / 连保底失败则通知 / 状态未知不重下）、guard ×1、LLM fail-closed ×2。

## 一句话总结

**代码负责安全和决策（硬限制写死、失败阶梯兜底、幂等防重扣），AI 只可选地帮忙读乱菜单且只能更保守，你平时只需要回「好 / 换 / 不要 / 不回也行」。核心：中午不会饿着，且不用动脑。**
